webtrees est écrit en anglais et utilise aussi l'anglais à l'interne. Le format de fichier GEDCOM utilise également des mots-clés et des identifiants anglais. Mais comme les humains parlent plusieurs langues, nous utilisons des fichiers de traduction pour convertir le texte en chaque langue. Les fichiers de traduction sont stockés dans le répertoire language/.

Notez que nous avons aussi des traductions pour les variantes régionales de la langue anglaise; d'où les traductions "Anglais Britannique" et "Anglais Américain".

Cette page explique comment vous pouvez personnaliser ces traductions en ajoutant un fichier de traduction personnel. Mais avant de commencer, vous devriez analyser pourquoi vous désirez effectuer cette action. Si c'est pour améliorer ou corriger une traduction, alors il est préférable d'offrir votre suggestion de traduction afin qu'elle soit incluse dans le fichier de traduction officiel et ainsi, d'en faire bénéficier toute la communauté. Pour ce faire, il suffit de se rendre sur translate.webtrees.net et de vous y inscrire. Vous pouvez aussi faire votre proposition sans ouvrir de compte ; dans les jours suivants, quelqu'un révisera votre suggestion et elle sera ensuite incluse dans la parution suivante.

Où stocker les fichiers de traduction personnels ?

Vous devez créer un sous-répertoire nommé "language" dans le répertoire data de webtrees, i.e. data/language/. Si vous avez déplacé votre répertoire data en-dehors du dossier-racine de votre serveur webtrees, alors il doit être créé dans votre nouveau répertoire data privé.

Formats et noms de fichiers

Les noms de fichiers se composent d'un code de langue et d'une extension indiquant le format. Exemples de codes de langues :

  • it (Italien)
  • fr (Français)
  • de (Allemand)
  • [email protected] (Serbe en script latin)
  • en_AU (Anglais australien)

Les trois formats qui peuvent être utilisés, sont décrits ci-dessous. Les traductions sont toujours de l'anglais vers la langue cible.

Format CSV

Le format le plus simple est CSV. Même si le CSV soutient les "valeurs séparées par des virgules", nous utilisons en réalité les point-virgules plutôt que les virgules, et tous les textes doivent être entourés par des guillemets. Voici un exemple d'un fichier CSV pour l'anglais britannique qui sera nommé data/language/en_GB.csv. 

"yes";"certainly"
"no";"perhaps not"

Ceci fera en sorte que le mot "yes" sera remplacé par le mot "certainly", partout où il se trouve dans le logiciel. Pour inclure des guillemets dans le texte, il faut les faire précéder de la barre oblique inversée "\".

Les avantages de ce format sont :

  • La simplicité. La création de fichier en utilisant un simple éditeur de texte ou un tableur.

Les inconvénients de ce format sont :

  • Ils ne peuvent être utilisés pour des traductions complexes, comme les pluriels (différentes langues ont différentes manières d'exprimer les pluriels, selon le nombre d'objets), ou les contextes multiples (où le même mot peut avoir plusieurs traductions, selon le contexte).
  • Selon la configuration de votre serveur, il peut être possible à un visiteur de votre site de voir le contenu du fichier, et que vous y ayiez inclus des informations sensibles.

Format PHP

Le deuxième format utilise un simple script PHP pour créer une chaîne de traductions. Le fichier contient non seulement les traductions mais aussi un simple code de vérification pour empêcher vos visiteurs d'accéder aux données. Voici un exemple d'un fichier PHP pour l'anglais américain qui sera nommé data/language/en_US.php. Pour inclure une apostrophe dans le texte, il faut la faire précéder de la barre oblique inversée "\". 

<?php
 if (!defined('WT_WEBTREES')) {
  header('HTTP/1.0 403 Forbidden');
  exit;
 }

 return array(
  'yes'=>'for sure!',
  'no'=>'no way!'
 );

Il est important de n'avoir aucune ligne en blanc ou d'espaces avant le code initial <?php. Le code final ?> est facultatif, alors nous ne l'incluons pas.

Les avantages de ce format sont :

  • Relativement simple. Vous pouvez créer les fichiers en utilisant un simple éditeur de texte.

Les inconvénients de ce format sont :

  • Ils ne peuvent être utilisés pour des traductions complexes, comme les pluriels (différentes langues ont différentes manières d'exprimer les pluriels, selon le nombre d'objets), ou les contextes multiples (où le même mot peut avoir plusieurs traductions, selon le contexte).

Format Gettext

Le troisième format est nommé gettext. Gettext est une librairie open-source, standard de l'industrie, pour les traductions. Chaque langue requiert deux fichiers. Il y a le fichier PO qui contient du texte lisible par les humains. Bien que vous puissiez modifier ces textes avec un éditeur de texte, un "éditeur PO" dédié est généralement utilisé. POEdit est un outil usuel, même s'il a un certain nombre de limitations; alors nous recommandons Better PO Editor, qui a été écrit par un des développeurs de webtrees, Michele Locati. Le fichier PO a besoin d'être transformé en fichier MO, qui est lisible par les machines, version compressée du même fichier. Mais il est possible que vous ayiez besoin du fichier binaire MO et de le convertir à nouveau en fichier PO afin de l'éditer avec le Better PO Editor. Car seulement POEdit possède l'exécutable msgunfmt pour faire cette conversion inverse. Alors, vous pourriez avoir besoin des deux outils.

Voici un exemple d'un fichier PO pour l'anglais américain, qui convertit certains termes juifs en orthographe yiddish. Le fichier sera nommé en_US.po et vous devrez le convertir en un fichier .mo qui sera nommé data/language/en_US.mo. Notez qu'il inclut plusieurs traductions selon le contexte, où vous pouvez avoir différentes traductions pour un même texte anglais. 

msgid ""
msgstr ""
"Project-Id-Version: webtrees\n"
"MIME-Version: 1.0\n"
"Content-Type: text/plain; charset=UTF-8\n"
"Content-Transfer-Encoding: 8bit\n"
"Plural-Forms: nplurals=2; plural=n != 1;\n"
"X-Poedit-Language: en\n"
"X-Poedit-Country: US\n"
"X-Poedit-SourceCharset: utf-8\n"

msgctxt "NOMINATIVE"
msgid "Heshvan"
msgstr "Cheshvan"

msgctxt "NOMINATIVE"
msgid "Tevet"
msgstr "Teves"

msgctxt "GENITIVE"
msgid "Heshvan"
msgstr "Cheshvan"

msgctxt "GENITIVE"
msgid "Tevet"
msgstr "Teves"

msgctxt "LOCATIVE"
msgid "Heshvan"
msgstr "Cheshvan"

msgctxt "LOCATIVE"
msgid "Tevet"
msgstr "Teves"

msgctxt "INSTRUMENTAL"
msgid "Heshvan"
msgstr "Cheshvan"

msgctxt "INSTRUMENTAL"
msgid "Tevet"
msgstr "Teves"

msgid "Heshvan"
msgstr "Cheshvan"

msgid "Tevet"
msgstr "Teves"

msgid "Bat mitzvah"
msgstr "Bas mitzvah"

msgid "Date of bat mitzvah"
msgstr "Date of bas mitzva"

msgid "Place of bat mitzvah"
msgstr "Place of bas mitzvah"

msgid "Source for bat mitzvah"
msgstr "Source for bas mitzvah"

Les avantages de ce format sont :

  • Il peut être utilisé pour toutes traductions, incluant les pluriels complexes et les différents contextes. C'est le format utilisé par webtrees pour ses propres fichiers de traduction.
  • Pour les fichiers contenant un grand nombre de traductions, c'est plus rapide et cela utilise moins de mémoire que les autres formats, et ne nécessite pas le téléchargement de toutes les traductions dans la mémoire.

Les inconvénients de ce format sont :

  • Il nécessite un logiciel additionnel pour créer les fichiers.

L'encodage des caractères

Les fichiers doivent être enregistrés en langage UTF8, plutôt qu'en ISO-8859, latin etc. Vous devez être tout spécialement vigilant avec certains éditeurs de texte qui débutent automatiquement les fichiers en ajoutant un caractère invisible (nommé un Byte-Order-Mark). Si, après avoir installé votre fichier, vous obtenez un message d'erreur comme "Headers already sent", vous avez probablement inclus un caractère invisible BOM.

Notes additionnelles

  • Peu importe la langue dans laquelle vous traduisez, votre fichier de langue personnalisé doit toujours être traduit à partir du texte du core code (I18N::translate('xxxxx'), et non pas à partir de n'importe lequel des fichiers standards de langue. Si vous désirez voir ces textes à l'écran, assurez-vous de les afficher en anglais en-US, plutôt que dans n'importe quelles autres langues.
  • La librairie des traductions utilise plusieurs caches. Alors, après avoir ajouté votre fichier personnalisé, vous devez vider le cache (du dossier « temp » de votre serveur), ou attendre au moins deux heures. L'endroit exact de votre cache et le temps requis de nettoyage automatique varient selon votre configuration.
