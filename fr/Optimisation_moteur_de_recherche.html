Optimisation moteur de recherche

De webtrees
Aller à : navigation, rechercher


Cette page nécessite webtrees 1.1 à minima.

La plupart d'entre nous publions nos données dans webtrees dans l'espoir que d'autres personnes découvrent nos recherches et nous fournissent des détails supplémentaires. Pour ce faire, nous avons besoin que les moteurs de recherche indexent le plus grand nombre possible de nos pages. webtrees 1.1 fait plusieurs avancées dans le domaine de l'optimisation du moteur de recherche (Search Engine Optimization => SEO), décrit ci-dessous. Notez que l'administrateur doit prendre plusieurs mesures pour cela.

Le contenu qui doit être indexé

Le contenu le plus important est constitué des données directes de notre arbre généalogique: ces pages doivent donc être les premières à être répertoriées (avec la page index.php): 

 individual.php, family.php, note.php, source.php, repo.php

Les listes correspondantes doivent également être indexées car elles fournissent les seuls liens complets vers les pages de données: 

 indilist.php, famlist.php, notelist.php, sourcelist.php, repolist.php

Les autres pages n'ont pas à être indexées. Elles fournissent une grande quantité de liens qui détournent les moteurs de recherche des pages clés. Par conséquent, webtrees 1.1 bloque l'indexation de celles-ci par les moteurs de recherche en utilisant les techniques énumérées sur cette page.

Note: les pages de listes (indilist.php, famlist.php, etc) ne sont pas très efficaces. En fait, l'arbre d'une famille nombreuse avait plus de 2000 versions de indilist.php indexées par Google! Si vous êtes en mesure d'utiliser le module Sitemap pour générer des liens de vos personnes et de vos familles, vous pouvez décider de bloquer indilist.php et famlist.php sur votre site en modifiant votre fichier robots.txt.

Suppression du contenu dupliqué

URL Canonical

wikipedia (pas de documentation en français trouvée pour l'instant)

Les moteurs de recherche n'aiment pas quand le même contenu est disponible via des URL différentes. Ainsi, sur les pages-clés que nous souhaitons qu'elles soient indexées webtrees 1.1 crée un lien vers la balise meta "canonical" comme ceci: 

 <link rel="canonical" href="family.php?famid=F6258&ged=xx.ged" />

Quand un moteur de recherche visite l'une de ces URL:

  • family.php?famid=F6258
  • family.php?famid=F6258&show_full=1
  • family.php?famid=F6258&show_full=1&ged=xx.ged
  • family.php?famid=F6258&ged=xx.ged

La balise meta canonical dit que c'est la seule version nécessaire pour l'indexation:

  • family.php?famid=F6258&ged=xx.ged

Ceci supprime les trois pages de contenu en double dans l'indexation par le moteur de recherche. Non seulement cela rend les moteurs de recherche heureux (et théoriquement plus de chances d'obtenir des liens vers votre site), c'est préférable pour les utilisateurs puisqu'ils n'ont pas à parcourir de nombreux résultats en double.

Aucune action n'est nécessaire pour profiter de cette amélioration.

Canonical et noms de domaines

Dans le prolongement de ce qui précède, les moteurs de recherche n'aiment pas quand le même contenu est disponible sous des domaines multiples ( par exemple www.example.com et example.com)

Action

Pour résoudre cela, allez dans l'écran de configuration du site et définissez l'URL préférée dans le champ "URL du site web" 

 Administration>Configuration du site>URL du site web (champ)

pour rediriger tous les accès à votre site vers la même URL.

Note: si ce paramètre est vide, attendre la mise à jour vers webtrees 1.1 avant de l'installer. Votre site est probablement déjà classé sous plusieurs domaines et les modifications dans la version 1.1 vont s'assurer que ces résultats de recherche soient réunis plutôt que perdus.

Dites aux moteurs de recherche ce qui est important

Les moteurs de recherche n'étudient pas les ressources pour indexer toutes les URL uniques sur un domaine, mais ils indexent tout ce qu'ils trouvent jusqu'à ce qu'ils décident d'arrêter. Nous ne pouvons pas contrôler quand ils choisissent d'arrêter, nous devons donc leur donner des indices sur ce que nous pensons important.

Il y a deux façons de s'y prendre:

  • Dites-leur précisément ce qu'ils NE doivent PAS indexer (en utilisant les en-têtes HTTP, les balises meta pour les robots, et le fichier robots.txt)
  • Dites-leur exactement ce qu'il faut indexer (en utilisant sitemap.xml)

Les en-têtes HTTP

Sur les pages bloquées par nos contrôles de confidentialité, webtrees renvoie désormais l'erreur 403 'Accès interdit'. C'est le moyen officiel pour indiquer aux moteurs de recherche, qu'ils n'ont pas accès au contenu. Cela n'a aucun effet sur la façon dont les navigateurs affichent le contenu aux utilisateurs.

Aucune action n'est nécessaire pour profiter de cette fonctionnalité.

Les balises meta pour les robots

Sur les pages qui doivent être indexées, webtrees définit maintenant précisément cette balise meta: 

 <meta name="robots" content="index,follow" />

Sur les pages qui ne doivent pas être indexées webtrees définit: 

 <meta name="robots" content="noindex,nofollow" />

Pour information, il y a aussi une possibilité intermédiaire qui signifie "suivre les liens sur la page mais ne pas indexer cette page": 

 <meta name="robots" content="noindex,follow" />

L'avantage d'utiliser ces balises META, c'est qu'elles sont très précises. Vous pouvez donc indiquer au moteur de recherche qu'il est autorisé à indexer family.php pour un arbre généalogique mais pas pour un autre.

L'inconvénient est que le moteur de recherche ne sait ce que vous voulez ignorer qu'après avoir déjà téléchargé la page; c'est un gaspillage de ressources.

Le fichier robots.txt

C'est semblable aux balises meta pour les robots, avec des atouts et des faiblesses.

L'avantage d'utiliser un fichier robots.txt est qu'il empêche le moteur de recherche de visiter la page avant de l'indexer.

L'inconvénient est que c'est tout ou rien: nous ne pouvons que dire oui ou non à l'indexation de family.php, nous ne pouvons pas le bloquer pour un arbre généalogique spécifique.

Dans le meilleur des cas, vous avez besoin d'utiliser les deux: le fichier robots.txt et les balises meta robots.

Davantage d'informations (en anglais). Davantage d'informations (en français).

Action

webtrees dispose d'un fichier appelé robots-example.txt. L'administrateur devrait le disposer à la racine du domaine. Il devrait donc être disponible à une adresse comme celle-ci: http://www.domaine.org/robots.txt

Toutes les lignes "Disallow" (Interdire) et "Allow" (Autoriser) doivent être adaptées pour prendre en compte le chemin de webtrees. Par exemple, la ligne Disallow:/admin.php devrait être remplacée par quelque chose comme Disallow:/webtrees/admin.php.

Un module complémentaire dynamique (Add On) est mis à disposition pour automatiser cette procédure à votre place.

Le fichier sitemap.xml

Il s'agit d'une liste de toutes les pages de votre site pour lesquelles vous souhaitez l'indexation par les moteurs de recherche. Il permet de donner la priorité au contenu (Par exemple, indexer individual.php, puis family.php, puis source.php, etc). Vous pouvez également spécifier la date de mise à jour d'une URL spécifique, ainsi les moteurs de recherche ne perdent pas leur temps à chercher les mises à jour sur les pages qui n'ont pas été modifiées.

  • Davantage d'informations sur sitemaps de Google (en français).

Action

webtrees inclut un module Sitemap qui fonctionne bien pour les arbres généalogiques de petite à moyenne taille, mais il sera probablement hors délai pour les grands arbres.

Optimisation du contenu

Une fois que les moteurs de recherche ont décidé de s'intéresser à votre page, certaines actions peuvent être faites pour les aider à comprendre ce qui est important. Nous n'avons pas évalué webtrees dans cette voie pour l'instant.

Moteurs malveillants

Notez que rien de tout cela ne permet de contrôler les robots malveillants, le but est d'essayer d'obtenir que le contenu le plus important soit indexé par les moteurs de recherche.
