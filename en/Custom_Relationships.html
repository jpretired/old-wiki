Custom Relationships

From webtrees
Jump to: navigation, search

The RELAtionship descriptor used in the ASSOciate tag is meant to be free-form text based on the GEDCOM 5.5.1 file format. But webtrees only supports a drop-down, selectable list of Relationships that are hard-coded into the source code. Therefore, one must actually change the underlying PHP code to add a Relationship not pre-defined in the list. Luckily, this is pretty easy to do even if you are not a coder as long as you have access to your installation and its source files. You may also need to edit the language files to support the newly added Relationships in other languages.

See the General guidelines for some cautions on customizing webtrees by modifying the source code. Specifically, your changes may get overriden by future updates or not be supported in the same way in future releases as the code evolves. In fact, webtrees added support for gender-context-sensitive 'Relationship titles after the fork from the original PHPGedView code base. This rendered previous changes obsolete and required one to re-code them in. See also Custom Tags for more warnings and a general overview. The way you add a custom tag is very similar to how you add a Relationship. But unlike custom tags, additional Relationships are encouraged and allowed for in the standard. It is just this tool implementation that limits them.

If you think your customizations belong in the core code, discuss this on the forum.

Modifying the Drop-down list

With that said, the changes needed to modify the list of Relationship titles in the drop-down is isolated to a single file in the installations source tree: app/GedcomCode/GedcomCode/Rela.php. Although a PHP source file, it has a clean structure that non-programmers should be able to easily modify without introducing errors. There are two main parts to this code of concern here. The $TYPE array definition of internal code names and the function getValue that converts the stored code name to text for display or output. Both areas must be edited to enable a new Relationship.

Before starting, make sure you have access to edit and replace the source file. Make a backup copy of the file in case you need to restore the original and to always have a clean copy to start from. Convention is to copy the file and rename it with an .old or .orig extension. So Rela.php.old in this instance.

Editing Rela.php $TYPES array

private static $TYPES=array(
  ....
);

The array is defined as a list of single-quote delineated names that are comma separated. As these are code names, there are no intervening spaces between the single quotes. Use an underscore instead of any space. Convention has it that the code name mimic the base, standard English name. While the entries do not have to be in any particular order, and the order in the array and the function do not have to match, good coding style dictates you keep them in ascending alphabetic order so perusing the file makes it easier to find. This is purely an internal name used by the PHP source code to find the language version to display. Code names in this array are case sensitive (maintain the same case in all uses).

There is no real limit as to how many comma-separated code names can be on a line. Convention would suggest you insert your additions into the list in order and put it on its own line with a comment following that indicates you added the code. So if we were to insert the code 'sponsor' in the existing line: 'slave', 'ward', 'witness', then the before and after may look like: 

BEFORE:
  'slave', 'ward', 'witness',
AFTER:
  'slave', 
  'sponsor', // Added by REH on 10/30/2011
  'ward', 'witness',

The double forward slash (//) indicates an uninterpreted text comment (like a GEDCOM note) until the end of the line.

Editing Rela.php getValue() function

webtrees is written in US English, and uses US English internally. So by default, text that is to become visible to users is coded with the English version as a base. webtrees supports many languages, some with context dependent gender dependencies,. So defining the text for the ASSOciate tag RELAtionships is a little more complicated than likely first imagined. Hence you will see some of the gender switch/case controls in the code base with some Relationship names.

The getValue() function is simply one large switch/case statement. Think of it as "pick one of". Given an array entry or code value it will generate a call to a text translation function for that value. So if you add an entry to the array above, you need to add a case entry point in the switch here.

There are two main types of entries here. Possible gender dependent names and unisex or fixed-gender names. Although not common in English, gender-dependent names are common in many other languages. But they are even in English. A parent is unisex whereas a father or mother is traditionally a fixed-gender variant. Similarly, a Godchild is unisex whereas a Goddaughter or Godson is fixed-gender. What is important here is not what is tradition or a possible exception you want to support, but what is the customary construct to use in other languages. If a language traditionally uses variants based on gender, support it in the code here.

So carrying on with our example above, the insertion for a sponsor in the getValue() switch would look like: 

case 'slave':
  switch ($sex) {
    case 'M': return I18N::translate_c('MALE',   'Slave');
    case 'F': return I18N::translate_c('FEMALE', 'Slave');
    default:  return I18N::translate  (          'Slave');
  }
case 'sponsor':  // Added REH 10/30/2011
  // no sex implied
  return I18N::translate('Sponsor');
case 'ward':
  switch ($sex) {

The call to function I18N::translate() is what performs the language translation and looks up the appropriate word in the language file. A variant of the call takes the gender as the first parameter.

Note that the parameter inside the parenthesis is the default, English language form of the Relationship intended and not the code value used in the case statement preceeding it. So while still enclosed in single quotes, spaces and alphabetic case are expected. So the code names in the array are used purely inside the PHP code switch statement to index the array. The name passed to the I18N::translate() function is the base English form that will be used to index the language files and find the translated version of. It is only this latter form that you will ever see in the user interface of the webtrees tool or in the GEDCOM file output from the tool.

Customizing the local language files

See the page Custom Translation about how to customize the local language files. Key is you should also add the new Relationships to the translation files as well as the source code. Beside the base language file, you can also add them to other languages as well. The default, if no language files are modified, is to simply use the English hard-coded in the source code for all output and display. So modifying the language file is not a requirement to get your change working but a necessary step if wanting to support other languages in your user interface.

Checking your changes

To see if you changes were successful, both in terms of adding the new Relationship as well as not breaking the tool and code base, you should always check the functionality changed.

Simply go to any Individual and add an ASSOciate fact or edit any EVENt such as a birth, death or similar and click to add an ASSOciate fact there. There are two parameters to enter, the actual ID of another Individual and the Relationship. For each language you changed, click the drop down box of the Relationship field to see that your added relationships are there (and that all the previous ones are still there). You may need to do this for different gender Individuals to fully check all versions. What appears here in the content from a language file, or if not translated, the original English text entered into the PHP code call to the translate function.

Use of the ASSOciate tag

The GEDCOM 5.5.1 standard defines the ASSOciate as a level 1 tag ONLY available in an INDIvidual record (top level). It is meant to associate an individual to another individual directly. In fact, the ASSOciate tag takes an INDIviduals xref_ID as a parameter. So this allows a processable way to link people other than through the FAMIly record.

Many tools, webtrees' included, also allow the ASSOciate tag inside EVENt tags in INDIvidual and FAMIly Records as well. Some even inside SOURce and other top level Records. Wherever allowed by the webtrees tool, the corresponding GEDCOM file output will show the ASSOciate tag at that place. So be aware that although associations are supported in these additional places in this tool, they may not be transportable to other tools except as a top-level tag in an Individual's record only.

The RELAtionship descriptor inside an ASSOciate tag is defined as a free-form text field and not a machine-interpreted name. So although hard-coded in webtrees, the pre-defined names used have no special meaning in the tool. It is purely a descriptor to allow the link to another Individual to be described. Because it is a descriptor, the list has been pre-defined so it can be properly translated into other languages in the interface.

The most common usage of the ASSOciate tag is in traditional religious events such as Baptism/Christening, Confirmation/BarMitzvah/Najote, and Marriage. The tag allows one to specify the godparents, sponsor and wedding party members. Especially for marriage, the list as defined in the tool today, is not complete for most traditions in common religions. Hence the need to modify what is allowed as a Relationship.

Note that ASSOciates are one way links. Meaning, if you specify a Godfather in a child (Individuals) Baptism record, there is no way to know this when looking at anything other than that child's record. You have to add an ASSOciate tag in the Godfather (Individuals) record to specify who the Godchild is to make the association bi-directional and visible from both Individuals.

Note also that although this is an alternative way to link Individuals outside of the Family link, most tools that process GEDCOM's do not recognize this link other than for display purposes. So the Relationship Charting tool in webtrees, for example, will not track through these associations. Nor will the clippings tool to extract a linked subset of your database.

Existing and Suggested Relationship values

webtrees has the following list of RELAtionships for the ASSOciate tag: 

'attendant', 'attending', 'best_man', 'bridesmaid', 'buyer',
'circumciser', 'civil_registrar', 'employee', 'employer', 'foster_child',
'foster_father', 'foster_mother', 'friend', 'godfather', 'godmother',
'godparent', 'godson', 'goddaughter', 'godchild', 'guardian',
'informant', 'lodger', 'nanny', 'nurse', 'owner',
'priest', 'rabbi', 'registry_officer', 'seller', 'servant',
'slave', 'ward', 'witness',

The PHPGedView code base, from which webtrees forked, also has defined: 

'twin', 'twin_brother', and 'twin_sister'.  However webtrees identifies these automatically.

Useful relationships one might consider adding, for example, are: 

'flower_girl', 'groomsman', 'maid_of_honor', 'matron_of_honor', 'officiant', 'ring_bearer', 'sponsor', and 'usher'

Sometimes the relationship is not really associated with an Event but should be top-level in an Individual or Family record. This is especially true with close friends that may have been more important than blood relatives. In this case, one can consider adding Relationships such as: 

'friend', 'close_friend', 'best_friend', 'family_friend', 'fiance', and the like.

It does not have to end there; just as with modern Social Network sites. You can allow all sorts of classifications or circles of people. So maybe you need: 

'colleague', 'classmate', 'war_buddy' and so on.

Key is this list is pre-defined and so you want to support the flexibility for researchers to capture information they collect about people with this more processable form. The ASSOciate relationship allows just that by letting a user navigate from Individual to Individual via the Relationship links defined.

Biggest downside to adding more Relationships is the list is presented as a drop-down menu. So the longer the list, the more trouble a user may have finding the relationship they are looking for. The list is not subset-ted based on the context of an event or record type you are adding the relationship too. So a Christening event will have the full list presented including Lodger, Seller, and the like.
