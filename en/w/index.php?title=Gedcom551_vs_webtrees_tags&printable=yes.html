The purpose of this page is to describe tags used in webtrees, that either do not exist in the standard GEDCOM 5.5.1, or may be interpreted in a different way. The standard can be found on this wiki's main page link.

GEDCOM 5.5.1 extended / exploded down to level number and tag

Reading the GEDCOM document is difficult, because of how it has been substructured. This file shows the level number that should be used with a tag, within a certain record or structure. The purpose of this file is to make clear how to use for example the address or place structure in the indidivual record (or others). Please inform me if you find errors. lexoulu

tag: _WT_OBJE_SORT

webtrees specific database tag. Not included in the GEDCOM 5.5.1 standard.

Description: Provides a control over the order of display on screen for media objects. This is a field required in the relational database to keep the data in a certain and proper sequence. The data in a GEDCOM standard file is presented sequentialy line by line. "...The occurrence of equal level numbers and equal tags within the same context imply that multiple opinions or multiple values of the data exist. The significance of the order in these cases is interpreted as the submitter's preference. The most preferred value being the first with the least preferred data listed in subsequent lines by order of decreasing preference. For example, a researcher who discovers conflicting evidence about a person's birth event would list the most credible information first and the least credible or least preferred items last..." page 20.

This tag is necessary in Object Schema - Data Schema mapping.

This tag is not to be included when exporting in GEDCOM format.


Occurs in structure: INDIVIDUAL_RECORD

tag: _WT_USER

webtrees specific tag. Not included in the GEDCOM 5.5.1 standard.

Description: Adds the user ID of the person who made the most recent change to this record in webtrees.

This tag is not to be included when exporting in GEDCOM format.


Occurs in structure:

<<CHANGE_DATE>>

CHANGE_DATE:=

n CHAN

+1 DATE <CHANGE_DATE>

+2 TIME <TIME_VALUE>

+2 _WT_USER

+1 <<NOTE_STRUCTURE>>

tag: ALIA

ALIA {ALIAS}:=

Description: An indicator to link different record descriptions of a person who may be the same person.

One or two systems used the ALIAs tag for representing multiple names. This form is not supported in the GEDCOM Standard version 5.5. page 69

GEDCOM 5.5.1 INDIVIDUAL_RECORD:= +1 ALIA @<XREF:INDI>@ {0:M} page.25

webtrees uses ALIA


ADDRESS_STRUCTURE

Description: The address structure should be formed as it would appear on a mailing label using the ADDR and the CONT lines to form the address structure. The ADDR and CONT lines are required for any address. The additional subordinate address tags such as STAE and CTRY are provided to be used by systems that have structured their addresses for indexing and sorting. For backward compatibility these lines are not to be used in lieu of the required ADDR.and CONT line structure.


GEDCOM 5.5.1 used in the following structures:

page 23 HEADER:= n HEAD +1 SOUR <APPROVED_SYSTEM_ID> +2 VERS <VERSION_NUMBER> +2 NAME <NAME_OF_PRODUCT> +2 CORP <NAME_OF_BUSINESS> +3 <<ADDRESS_STRUCTURE>>

page 27 REPOSITORY_RECORD:= n @<XREF:REPO>@ REPO +1 NAME <NAME_OF_REPOSITORY> +1 <<ADDRESS_STRUCTURE>>

page 28 SUBMITTER_RECORD:= n @<XREF:SUBM>@ SUBM +1 NAME <SUBMITTER_NAME> +1 <<ADDRESS_STRUCTURE>>


INDIVIDUAL_RECORD (page 25)-> +1 INDIVIDUAL_EVENT_STRUCTURE (page 25/34) -> +1 INDIVIDUAL_EVENT_DETAIL (page 34) -> n EVENT_DETAIL (page 32)-> n <<ADDRESS_STRUCTURE>>

INDIVIDUAL_RECORD (page 25)-> +1 INDIVIDUAL_ATTRIBUTE_STRUCTURE (page 25/33)->+1 INDIVIDUAL_EVENT_DETAIL (page 34) -> n EVENT_DETAIL (page 32)-> n <<ADDRESS_STRUCTURE>>

First tag in <<ADDRESS_STRUCTURE>> and is mandatory, n ADDR <ADDRESS_LINE>

IMPORTANT!

ADDR can never have level 1 in the INDIVIDUAL_RECORD (when INDIVIDUAL_RECORD has level 0)

If level 1 ADDR is encountered in the INDIVIDUAL_RECORD data, it should be considered as an error.


EVEN

Description: (page 35) The EVEN tag in this structure is for recording general events that are not shown in the above <<INDIVIDUAL_EVENT_STRUCTURE>>. The event indicated by this general EVEN tag is defined by the value of the subordinate TYPE tag. For example, a person that signed a lease for land dated October 2, 1837 and a lease for equipment dated November 4, 1837 would be written in GEDCOM as::

1 EVEN

2 TYPE Land Lease

2 DATE 2 OCT 1837

1 EVEN

2 TYPE Equipment Lease

2 DATE 4 NOV 1837


SOURCE_RECORD:= n @<XREF:SOUR>@ SOUR +1 DATA +2 EVEN <EVENTS_RECORDED>

SOURCE_CITATION (page 39) -> +1 EVEN <EVENT_TYPE_CITED_FROM>

FAMILY_EVENT_STRUCTURE:= -> n EVEN [<EVENT_DESCRIPTOR> | <NULL>] (page 32)

INDIVIDUAL_RECORD:= n @XREF:[email protected] INDI (page 25)-> INDIVIDUAL_EVENT_STRUCTURE:= (page 34) -> n EVEN (page 35)


IMPORTANT! 

n EVEN  in INDIVIDUAL_RECORD
 EVEN never has any line_value after the tag EVEN.

_NMR

_NMR is a *family* tag - it means a couple is not married.

_NMAR

_NMAR is an *individual* tag - it means the individual never married.
